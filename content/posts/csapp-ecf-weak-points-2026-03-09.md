今天简单过了一轮 CSAPP 里“异常控制流（Exceptional Control Flow）”这一章，没有往后推进太多，但已经能比较明确地看出目前的薄弱点在哪里。

这篇先不追求讲完整知识，而是把眼下最容易混淆的地方写下来，方便下一次继续补。

## 这次复习暴露出的主要薄弱点

### 1. trap 和 fault 的区分还不牢
目前最大的问题，不是完全不知道这两个词，而是知道得有点“浮”。

我已经能比较稳定地区分：

- interrupt 是异步的
- interrupt 往往来自外部事件或外设

但一到 trap 和 fault，就容易开始混。

现在能确认的正确方向是：

- trap 是同步异常
- trap 往往是有意触发的，比如 system call
- fault 也是同步异常
- fault 往往是当前指令执行时遇到了问题

真正还没完全钉死的，是这两者在“触发机制”和“处理后返回位置”上的差别。

### 2. 异常分类标准还没有彻底统一
这次问答里一个很明显的问题是：判断异常类别时，我会在多个标准之间来回切换，比如：

- 外部 / 内部
- 用户态 / 内核态
- 是不是系统服务
- 是不是出错

但更稳的判断框架应该是：

- 它是同步还是异步
- 它是外部事件打断，还是当前指令导致
- 它是主动触发还是被动触发
- 处理完之后通常回到当前指令还是下一条指令

这说明我现在不是“完全不会”，而是判断维度还不稳定，所以一旦问题稍微换个问法，就容易被绕进去。

### 3. 容易把“异常类型”和“处理结果”混在一起
这也是这次暴露出来的一个关键问题。

比如 page fault：

一开始很容易下意识把它理解成“访问出错，所以最后程序可能挂掉”，但这其实把两件不同的事揉到一起了：

1. 这个异常属于哪一类
2. 内核最后怎么处理它

更准确地说：

- page fault 的分类是 fault
- 它之所以是 fault，是因为它由当前指令触发，而且通常是可以修复后重试的
- 但如果地址非法、权限不对，最后也可能导致进程终止

也就是说：

> fault 是分类，终止是处理后果，这两个不是一个维度。

这个点如果不分清，后面继续学 signal、process、shell 的时候会越来越乱。

## 这次已经稍微清楚一点的地方

虽然这一章还没吃透，但也不是完全没进展。

至少这次已经开始形成几个初步的判断：

- interrupt 是异步的，而且来自外部
- page fault 不属于 interrupt，因为它不是外部打断，而是当前指令访问内存时触发的同步异常
- system call 不属于普通函数调用，因为它不是单纯在用户程序内部跳转，而是要借助 trap 进入内核态请求系统服务

这些点还不能说已经特别稳，但至少方向已经开始对了。

## 下一次复习最该先补什么

下一次不用急着往 signal 往后冲，先把异常分类这一小块真正钉死。

我最需要先彻底搞清楚的是下面这四句话：

1. interrupt 是异步的，通常来自外部设备或时钟
2. trap 是同步的、主动触发的，常用于 system call
3. fault 是同步的，由当前指令触发，通常可以修复并重新执行当前指令
4. abort 往往对应严重错误，通常不会返回

如果这四句话能彻底稳定下来，后面再学 signal、进程、shell，理解会顺很多。

## 现在的状态

所以这次复习的结论，不是“这一章已经掌握了”，而是：

> 已经知道目前最薄的地方在哪里，下一次应该继续围绕 trap、fault、page fault、system call 这几个点往下打。

这其实也挺好。比起模模糊糊觉得“差不多懂了”，更有用的是知道自己到底卡在哪。
